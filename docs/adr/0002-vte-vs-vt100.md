# 0002. Parser-only VT tracking via `vte`

- **Status:** Accepted
- **Date:** 2026-03-01 (retrospective; design predates this ADR)
- **Supersedes:** —
- **Superseded by:** —

## Context

Running as a PTY proxy (see [ADR-0001](0001-pty-proxy-vs-plugin.md)) forces us
to track cursor state from the shell's raw byte stream. There are three
mainstream Rust options for consuming VT / ANSI escape sequences:

1. **`alacritty_terminal`** — a full terminal emulator crate. Maintains a
   complete cell grid, scrollback buffer, alt-screen, selection, etc. Used as
   the rendering backend inside Alacritty proper.
2. **`vt100`** — a slimmer terminal emulator crate that still maintains a
   screen grid, intended for recording / replay tools.
3. **`vte`** — a parser-only state machine. Implements Paul Williams' VT500
   parser and fires user callbacks per escape sequence via the `Perform`
   trait. It does **not** own a grid, cursor, or screen state.

Ghost Complete needs exactly four things from the byte stream
(`CLAUDE.md:40`, `docs/ARCHITECTURE.md:110-117`):

- Cursor position (row, column)
- Screen dimensions (for popup layout)
- Prompt boundaries (via OSC 133 / OSC 7771 markers)
- Current working directory (via OSC 7)

We do not need a cell grid. We never replay scrollback, do not render cells,
do not handle selection, and do not care about most SGR state. The terminal
emulator downstream of us owns all of that.

## Decision

Use **`vte` with a custom `Perform` implementation** in `gc-parser`. Maintain
only the minimal state we actually query (cursor row/column, screen size,
prompt-start markers, CWD). Every byte that passes through the proxy is fed
to the parser; interesting callbacks mutate our tiny state struct and then we
forward the original byte untouched to the terminal.

`gc-parser` also owns the CPR (Cursor Position Report) reconciliation queue:
we periodically emit `CSI 6n`, the terminal answers with the real cursor
position, and we correct any drift our parser accumulated from sequences it
under-models. The CPR queue is strictly owner-tagged (`Ours` vs `Shell`, see
v0.8.1 `CHANGELOG.md`) so Atuin and other PTY programs' CPR requests are not
swallowed.

## Consequences

### Positive

- **Cheap parsing.** No grid allocations, no scrollback ring buffer, no
  per-cell attribute mask. Memory idle target (`<10MB`) is achievable — we
  hit `~8MB` in practice (`docs/ARCHITECTURE.md:186-191`).
- **Correctness where it matters.** `vte` implements the well-tested VT500
  parser state machine; we do not re-implement escape-sequence dispatch. Our
  bug surface is the `Perform` callbacks, which are small and unit-testable.
- **Forwarding is pass-through.** Because we do not materialise a grid, we
  never need to "re-render" the shell's output — we forward the original
  bytes unchanged and only inject popup sequences between them. This
  preserves every behaviour the downstream terminal already handles
  (Sixel, Kitty graphics, Kitty keyboard protocol, hyperlinks, etc.) without
  us needing to model them.
- **Fast benchmarks.** `vt_parse_throughput` Criterion benches
  (`CLAUDE.md:100`) stay in the sub-microsecond-per-sequence range.

### Negative

- **Cursor drift is our problem.** Any escape sequence we do not model
  correctly — in particular, margin-aware scrolling, right-to-left text, or
  exotic DECSET modes — causes our cursor tracker to drift from the
  terminal's real cursor. Mitigation: CPR reconciliation (`CSI 6n`), plus
  defensive clamping on terminal resize (v0.7.1 `CHANGELOG.md`).
- **No grid means no "read back" of shell output.** We cannot answer
  questions like "what text is currently on row 5?" If a feature needs that,
  we would either have to add a grid (expensive) or ask the terminal via an
  OSC query.
- **We hand-roll prompt-boundary logic.** A full emulator library might give
  us hooks for "an OSC landed" for free; with `vte` we wire it ourselves.
  This is a small cost and has been stable in practice.

## Alternatives considered

- **`alacritty_terminal`.** Rejected as vastly over-scoped. We would pay for
  a full grid on every keystroke to extract four scalars. Memory budget would
  miss the `<10MB` idle target by an order of magnitude.
- **`vt100`.** Still maintains a grid; smaller than `alacritty_terminal` but
  same category of overkill. Also less maintained than `vte`.
- **Write our own parser.** Rejected — Paul Williams' state machine is subtle
  and well-tested elsewhere. Owning the `Perform` callbacks is enough custom
  surface; we do not also need to own the tokenizer.
