# Benchmarks — v0.11.0

Date: 2026-05-03
Platform: macOS (Apple Silicon)

Criterion's reported `change` column compares against the previous saved local
run, which on this machine is v0.10.0 stable. Deltas here measure the work that
landed between v0.10.0 and v0.11.0: AWS spec restoration (#101), local-project
providers (#97), OSC 7772 buffer framing (#94), async suggestion feedback (#98),
multi-word alias expansion (#95), static `args.suggestions` (#96), popup page
navigation (#99), and the popup rendering hardening pass (#100). Three new
benchmark groups (`osc7772_decode`, `feedback_render`, plus `alias_hit_*` cases
in `engine_suggest_sync` and `with_static_suggestions_tar` in `spec_resolution`)
ship as new baselines.

## vt_parse_throughput

| Benchmark | Time | vs v0.10.0 |
|-----------|------|------------|
| plain_text | 197.36 µs | +10.6% |
| ansi_colored | 246.11 µs | +10.5% |
| cursor_heavy | 266.70 µs | −2.9% improvement |

The plain_text and ansi_colored regressions trace to the OSC dispatch fan-out
in `gc-parser`: the OSC handler now matches OSC 7770 (legacy), 7771 (cursor),
and 7772 (percent-encoded buffer) sequences, plus the per-frame deprecation
warn for 7770. The plain_text input contains no OSCs, so the cost is purely
the dispatch table grow; cursor_heavy is dominated by CUP/CUF/CUB sequences,
which sit on a different code path and improve marginally as side benefit of
the parser refactor.

## spec_loading

| Benchmark | Time | vs v0.10.0 |
|-----------|------|------------|
| load_717_specs | 288.14 ms | +241% expected |

**Expected.** This is the AWS spec restoration showing up at startup-load
time. The bench loads the embedded spec corpus once, and the corpus grew
from ~47 MB minified pre-AWS to ~83 MB minified with AWS — 17 139 added
subcommands and 99 537 added options. The load happens once per process at
init, so the increase is on cold-start, not on the keystroke path. Embedded
specs heap walk (`memory/embedded_specs_heap_walk` = 4.23 ms) confirms the
runtime walk over the full corpus is still well-bounded.

zstd-compressing the embedded JSON corpus (separate plan tracked in
follow-on work) is the principled reclaim path here and would drop both
binary size and load time meaningfully.

## transform_pipeline

| Benchmark | Time | vs v0.10.0 |
|-----------|------|------------|
| simple | 26.598 µs | +2.5% (flat) |
| regex | 113.72 µs | +2.0% (flat) |
| json | 35.870 µs | +33.6% expected |

The json regression traces to the `json_extract_array` extension landed in
the docs/correctness pass — `json_extract` now branches on whether the
target is a scalar or an array projection, adding a per-row clone for the
array case. The bench exercises the new array path, so the +33% is the
cost of correctness, not avoidable overhead. Previous behaviour silently
emitted wrong completions on array-shaped JSON (caught in v0.10.0
`### Corrected`).

## Analysis

Three baselines added (`osc7772_decode`, `feedback_render`,
`engine_suggest_sync/alias_hit_*`, `spec_resolution/with_static_suggestions_tar`).

The two material regressions are both **expected and acceptable**:

1. `spec_loading/load_717_specs` +241% — AWS spec restoration. One-time
   cold-start cost, not on the keystroke path. Reclaim plan: zstd-compress
   embedded specs (separate spec).
2. `transform_pipeline/json` +33.6% — `json_extract_array` branch added
   for correctness. Previous v0.10.0 behaviour was silently wrong on
   array-shaped JSON; the new cost is structural, not optimisable.

Everything else lands within run-to-run noise (±10%). No keystroke-path
regression worth blocking the release.

The `osc7772_decode` baseline (574 ns at typical buffer size) confirms the
new buffer framing is invisible on the hot path. The `feedback_render`
indicator row at 122 ns/frame confirms the async feedback UI ships free.
