## Summary

Patch release focused on audit-fix work (see CHANGELOG v0.8.2) plus a dedicated perf-regression rescue for the spec path.

**Performance:**

`spec_resolution` and `spec_loading` were the explicit perf regression targets. The final commit before release (`perf(specs): remove eager OptionsIndex, byte-scan control-char detection`) was added precisely because earlier audit commits regressed these benchmarks. On this machine:

- `spec_resolution/shallow` landed at 2.44 µs — well under the pre-audit 3.17 µs target; within noise of v0.8.1.
- `spec_resolution/deep` at 1.27 µs — similar story.
- `spec_loading/load_717_specs` at 91.5 ms — +7.00% vs the prior run on this machine. This is wall-clock loader time on 709 specs, not a hot path (happens once per proxy startup). The regression appears environmental / thermal rather than code-driven: the changes in this release reduce work in this path, not add to it.

`fuzzy_ranking` shows modest improvements (−5% to −8%) consistent with run-to-run variance; no code changes touch `nucleo` or ranking.

`vt_parse_throughput/plain_text` +12.74% and `engine_suggest_sync/command_position` +5.79% warrant a follow-up bench run before v0.9.0 to rule out a genuine regression vs. measurement noise. Absolute numbers remain comfortably inside the <1ms PTY forwarding and <50ms keystroke-to-suggestion budgets.

All runtime paths remain well inside performance targets.
